# Backend QA Report - Rich Aroma OS

## 1. Security Vulnerabilities

**Critical severity:**
*   **Test Backdoor Active:** `server.js` contains a hardcoded `TEST_TOKEN_ADMIN` backdoor in the `requireAuth` middleware. This grants full admin privileges and must be removed before production.
*   **Plaintext Passwords/PINs:** The `customers` table and `server.js` handle and store PINs and passwords in plaintext (`pin`, `password`). These must be cryptographically hashed (e.g., using bcrypt or Argon2) before storage.
*   **Public Sensitive Endpoints:** Routes like `/api/customer/test-cash` (adds 500 cash balance) and `/api/orders` (POST) are fully public and unauthenticated. A malicious user could drain inventory or print fake orders indefinitely.
*   **Open RLS Policies:** `supabase-schema.sql` sets `CREATE POLICY "Allow all" ON [table] FOR ALL USING (true);` for all tables. Anyone with the Supabase Anon Key can read, modify, or delete the entire database directly.

## 2. Race Conditions & Data Integrity

**High severity:**
*   **Order Number Generation:** In `POST /api/orders`, the order number is generated by querying the max order number and adding 1 (`const orderNum = (maxOrder?.[0]?.order_number || 0) + 1;`). Under concurrent load, multiple requests will read the same max number and attempt to insert the same ID, causing primary key conflicts or data loss. 
    *   *Fix:* Use the `order_number_seq` sequence defined in the SQL schema directly within an insert query, or use a Supabase RPC function to handle atomic increments.
*   **Customer ID Generation:** Similar to orders, `POST /api/auth/register` and `POST /api/campaign/valentines` manually calculate the next Customer ID by finding the max ID. This will collide under concurrent signups.
*   **Live Drop Inventory:** In `/api/live/claim` and `/api/live/pay`, `currentDrop.stock` is decremented in application memory. If the server scales horizontally (multiple instances) or restarts, stock data is lost or oversold.

## 3. Unhandled Promise Rejections & Error Handling

**Medium severity:**
*   **Silent Database Failures:** Many Supabase insertions and updates lack `.error` checks. For example, in `/api/live/pay`: `await supabase.from('balance_history').insert(...)` and `await supabase.from('orders').insert(...)` do not check for or return errors. If the database is unreachable, the API will still return `success: true`.
*   **Missing `try...catch` Blocks:** Many asynchronous routes lack a wrapping `try...catch` block. If `supabase.from()` throws an exception (e.g., network error), it will result in an unhandled promise rejection and could crash the Express server.
*   **Inventory Deduction Engine:** In `POST /api/orders`, if the `supabase.rpc('decrement_inventory', ...)` fails, the error is caught and logged, but the order is still processed. Depending on business requirements, inventory failures might need to alert staff or rollback the order.

## 4. Schema Mismatches

*   **Missing Columns in `customers`:** 
    *   The backend writes to a `tags` array (e.g., `tags: ['coloring_contest']` and in `notify` route).
    *   The backend writes a `referral_code` string.
    *   Neither of these columns exist in the `customers` table definition in `supabase-schema.sql`.
*   **Unused Tables/Sequences:** 
    *   `order_number_seq` is created but never used by the backend.
    *   `business_settings` and `loyalty_settings` are created but have no corresponding backend routes to manage them.

## Recommendations for Optimization
1.  **Atomic Transactions:** Use Supabase RPCs (PostgreSQL functions) for critical multi-step operations (e.g., deducting balance and creating an order) to ensure atomicity.
2.  **Lock Down RLS:** Immediately replace the `USING (true)` RLS policies with proper role-based access control tied to Supabase Auth (`auth.uid()`).
3.  **Use UUIDs:** Consider using UUIDs for `id` columns instead of sequential `TEXT` IDs (like `C001`, `ORD-0001`) to eliminate the need for max-ID lookups and prevent enumeration attacks.
